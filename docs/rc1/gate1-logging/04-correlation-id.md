# 04 — Correlation ID

## Current State

Two header names were in use:

- `x-request-id` — generated by the client in `utils/api.ts`.
- `x-correlation-id` — read by server `request-context.ts`.
- `x-runtime-correlation-id` — read by `runtime-observability.ts`.

This created two separate correlation namespaces.

## Remediation

1. **Client propagation**
   - Updated `apps/web/src/utils/api.ts::appendDefaultClientHeaders` to set both
     `x-request-id` and `x-correlation-id` to the same generated UUID.

2. **Server acceptance**
   - Updated `apps/web/src/lib/server/runtime-observability.ts::getRuntimeCorrelationId`
     to accept `x-runtime-correlation-id` first, then fall back to
     `x-correlation-id`, then generate a new ID.

3. **Structured logs**
   - Every `RUNTIME_EVENT` now carries:
     - `requestId` (from `x-request-id`)
     - `runtimeCorrelationId` (from correlation header chain)

## Propagation Matrix

| Layer | Before | After |
|-------|--------|-------|
| Browser client | `x-request-id` only | `x-request-id` + `x-correlation-id` |
| Next.js server | `x-correlation-id` only | `x-runtime-correlation-id` or `x-correlation-id` |
| Audit chain | correlation passed via `writeAuditLog` | unchanged (already receives request) |
| Background jobs | no correlation | can pass `runtimeCorrelationId` to `logRuntimeEvent` |
| CKE / PDF / OTP | no correlation | now inherit from request context |

## Verification

- New unit tests confirm:
  - `x-runtime-correlation-id` is preferred.
  - `x-correlation-id` is used as fallback.
- Full test suite and build pass.

## Recommendation

Adopt `x-correlation-id` as the single canonical header across all internal
services. The current dual-header support is a compatibility shim and can be
removed in a future cleanup sprint.
