# 22 — Login Refactor Verification (VE-02)

## What Changed

The login page (`apps/web/src/app/login/page.tsx`) was refactored to consume the new design-system components while keeping the existing authentication logic intact.

### DS Components Now Used on Login

- `Container` — outer page grid wrapper
- `Section` — brand and form panels
- `Stack` — vertical feature list
- `Card`, `CardHeader`, `CardTitle`, `CardDescription` — login card
- `Form` — password submission form
- `FormField` — labeled email/password fields
- `Input` — email and password inputs with start/end icons
- `Button` — primary sign-in and secondary IMC buttons
- `Alert` — inline credential-error message

### Preserved Behavior

- Password login API call (`/api/auth/password/login`)
- `rememberMe` checkbox state
- `mustChangePassword` redirect to `/first-login`
- `next` query-param redirect handling via `normalizePostLoginDestination`
- RTL detection and Arabic font application

### Styling Approach

Because `styled-jsx` scoped classes are not applied to custom component roots, and because `<style jsx global>` is not emitted into production builds in the App Router, the page-specific overrides were moved to a route-scoped stylesheet.

- `apps/web/src/app/login/login.css` — all login visual overrides (card radius, input height, button gradients, brand panel orbs, etc.).
- `apps/web/src/app/login/layout.tsx` — imports `login.css` so the styles are loaded only for `/login` and do not leak to the rest of the app.

The page component (`apps/web/src/app/login/page.tsx`) no longer contains any `<style jsx>` block.

### Screenshot Comparison

Screenshots were captured with Playwright Chromium at desktop (1280×900) and mobile (390×844) viewports for both English and Arabic. They were compared against the approved VE-1 baseline in `qa-screenshots/login-ve1/` using the diff utility at `scripts/capture-login-ve2.mjs`.

The utility compares the overlapping region when heights differ, using a perceptual color-distance threshold of 30.

| Viewport | Baseline Size | Current Size | Mismatch (overlapping area) | Status (15% tolerance) |
|----------|---------------|--------------|----------------------------|------------------------|
| Desktop EN | 1280×936 | 1280×900 | 11.459% | ✅ PASS |
| Desktop AR | 1280×900 | 1280×900 | 12.349% | ✅ PASS |
| Mobile EN | 390×1394 | 390×1379 | 24.536% | ❌ FAIL |
| Mobile AR | 390×1342 | 390×1346 | 21.077% | ❌ FAIL |

### Desktop Assessment

Desktop English and Arabic both pass the 15% visual-tolerance threshold. The remaining 11–12% diff is almost entirely anti-aliasing, sub-pixel orb/gradient placement, and minor font-metrics differences inherent to rendering DS-generated markup versus the original hand-written markup. No functional or layout regressions are visible.

### Mobile Assessment

Mobile screenshots are close to the baseline in overall height (within ~1%) but exceed the strict 15% pixel-diff tolerance. The elevated mismatch is driven by:

- **Baseline inconsistencies**: the VE-1 mobile English baseline shows the language pill as `English`, while the VE-1 desktop English baseline shows `EN`; the current code path renders `EN` consistently and cannot produce both values.
- **Compact-viewport wrapping**: the brand-panel headline and feature boxes wrap differently on the narrow 390 px viewport because the markup is now generated by DS components with their own box-model defaults; the page was tuned with mobile-specific compact styles but remains a few pixels taller/shorter in places.
- **Sub-pixel layout shifts**: icon sizing, feature-box padding, and orb gradients differ by a handful of pixels from the hand-written baseline.

No content is missing and the mobile experience remains usable and on-brand. The desktop viewport, which is the primary login use case, matches the baseline comfortably.
