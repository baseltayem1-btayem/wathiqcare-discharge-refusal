# Backend Architecture

## Scope
This document describes the production-oriented NestJS backend foundation implemented in `backend-nest` for discharge-refusal case management.

## Technology Stack
- Runtime: Node.js + NestJS + TypeScript
- Database: PostgreSQL
- ORM: Prisma
- Cache/ephemeral verification state: Redis
- Object storage target: S3-compatible endpoint (MinIO in local development)
- Auth: JWT access/refresh token pair
- API docs: Swagger UI (`/api/docs`)

## Architecture Style
- Modular monolith with explicit domain modules and shared infrastructure.
- Global concerns (validation, logging, response envelope, exception handling) are centralized and applied once at bootstrap.
- Tenant-aware execution model: services scope reads/writes by `tenantId` for non-global records.

## Bootstrap and Cross-Cutting Components
- `src/main.ts`
  - Helmet + compression middleware
  - Global `ValidationPipe` with whitelist and type transformation
  - Global exception filter: normalized API error shape
  - Global interceptors: request logging and response envelope
  - Swagger initialization
- `src/config/app.config.ts`
  - Environment-driven config (DB, JWT, Redis, S3, defaults)
- `src/prisma/*`
  - Prisma client lifecycle and graceful shutdown hooks

## Request Path (Typical)
1. Incoming request carries bearer token.
2. JWT strategy validates identity and active status.
3. User roles/permissions are hydrated for request context.
4. Permission guard enforces action-level authorization.
5. DTO validation sanitizes payload.
6. Service performs tenant-scoped business logic, often in Prisma transaction blocks.
7. Domain mutation is logged in audit trail.
8. Response is wrapped in consistent envelope format.

## Deployment Notes
- Build artifacts generated by `nest build` and Prisma client generation.
- Prisma migration deployment is expected at startup/deploy phase.
- Runtime requires reachable Postgres, Redis, and S3-compatible object storage endpoint.
